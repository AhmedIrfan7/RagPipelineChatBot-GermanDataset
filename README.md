# German Driving-School Chatbot

A production-grade, bilingual (German and English) assistant that answers pricing and
enrollment questions for a German driving school. It is built around one non-negotiable
principle: **it must never give a wrong price.**

Prices are never generated by a language model. They are read verbatim from a
deterministic, human-verified structured store and checked against the original source
PDFs. The language model is used only for two narrow, bounded tasks: routing free text
that the rule-based layer cannot resolve, and translating the client's own text into
English. Nothing about pricing depends on the model.

For the full story (what the system does, why it is designed this way, and how every
component works), read [`docs/PROJECT_DOCUMENTATION.md`](docs/PROJECT_DOCUMENTATION.md)
(also available as a formatted [PDF](docs/PROJECT_DOCUMENTATION.pdf) with an
architecture diagram).

## What it does

- **Exact pricing with a full breakdown.** For any license class and variant, returns
  the exact total, an itemized cost breakdown, a note about external fees the school
  does not charge, and the official price sheet as a downloadable PDF.
- **Guided follow-up questions.** When a request could mean several variants (a car
  license alone has 18), the bot asks short, clear questions until exactly one variant
  remains, then prices it.
- **General question answering.** Opening hours, required documents, document validity,
  simulator availability, registration steps, answered from the client's own written
  material with a precision-first retrieval policy (it hands off rather than guessing).
- **Bilingual, automatically.** Detects German or English and mirrors the language,
  including faithful translation of German information into English.
- **Safe fallback.** When it is not confident, it connects the customer to the team
  instead of guessing.

## Architecture

```
Customer message (German or English)
        |
Language detection -> Intent and class detection (rules first)
        |
        +-- price intent --------> Resolve license class --+
        |                                                    |
        +-- general question ----> Knowledge retrieval       +-- one variant --> Structured price store --> Price card + PDF
        |         |                                          |
        |         +-- confident --> Answer from client text  +-- ambiguous ----> Disambiguation tree (follow-up questions)
        |         +-- not confident --> Hand off to team
        |
        +-- unclear free text ---> LLM router (validated class only) --+
                                                                          (feeds into Resolve license class above)
```

Every euro amount travels the `Resolve -> Disambiguation/Store -> Price card` path. It
is read from the verified store and never generated by the model. A misunderstanding can
lead to another question or a hand-off; it can never produce a wrong number. See the
[full diagram](docs/PROJECT_DOCUMENTATION.pdf) for the complete flow with edge labels.

| Layer | Responsibility |
| --- | --- |
| `src/fahrschule/taxonomy.py` | German license-class knowledge; uncertain codes flagged, never guessed |
| `src/fahrschule/store.py` | Deterministic price store and query layer (`get_price`, `list_variants`, `resolve_class`, `get_document_link`) |
| `src/fahrschule/disambiguation.py` | Follow-up decision trees that narrow an ambiguous class to one variant |
| `src/fahrschule/knowledge.py` | Hybrid FAQ retrieval (keyword search re-ranking semantic candidates), precision-gated |
| `src/fahrschule/embeddings.py` | Semantic index (cosine similarity over cached OpenAI embeddings) |
| `src/fahrschule/llm.py` | Bounded language-model adapter: validated-class routing and faithful translation only |
| `src/fahrschule/dialogue.py` | Conversation engine: language detection, intent routing, session memory, reply formatting |
| `src/fahrschule/api.py` | FastAPI service and embeddable web chat widget |

## Confidentiality

The client's documents are confidential and contain business and financial details. The
entire `data/` tree, and everything derived from it (extracted prices, the knowledge
base, the embeddings cache), is excluded from version control and never published. Only
application code, schemas, tests, and documentation are public. Secrets (the OpenAI API
key, the hand-off email, the business name) are read from a local `.env` file that is
never committed. One exception is documented in `.gitignore`: the project documentation
PDF, which is verified content-free of prices or client data before every commit.

## Getting started

### 1. Prepare the data (one-time, offline, requires the client PDFs)

```bash
python -m venv .venv
.venv/Scripts/pip install -r requirements.txt      # Windows
# .venv/bin/pip install -r requirements.txt        # macOS / Linux

# place the client PDFs in data/raw/, then:
python scripts/01_build_manifest.py
python scripts/03_extract_prices.py
python scripts/05_extract_knowledge.py
python scripts/04_build_store.py
```

### 2. Configure secrets

```bash
cp .env.example .env
# fill in OPENAI_API_KEY, HANDOFF_EMAIL, SCHOOL_NAME
```

### 3. Run the tests

```bash
python -m unittest discover -s tests
```

### 4. Run the service

```bash
.venv/Scripts/python.exe -m uvicorn fahrschule.api:app --app-dir src --port 8123
# open http://localhost:8123/
```

### 5. Or run it in Docker

```bash
docker compose up --build
# open http://localhost:8123/
```

See [`docs/DEPLOY.md`](docs/DEPLOY.md) for hosting guidance.

## Testing and quality assurance

Validated at three levels, then attacked on purpose:

1. **80 automated tests** covering the taxonomy parser, the price store and its
   resolution logic, the disambiguation trees, both language flows, retrieval, the
   language layer (with a fake model, so tests make no live calls), and the web API.
2. **An evaluation gate** (`tests/test_eval_regression.py`, `scripts/06_eval.py`) that
   runs the client's own example questions plus adversarial probes end to end.
3. **A ground-truth price gate** (`tests/test_price_vs_pdf.py`) that re-parses the grand
   total directly from each source PDF with a second parser and compares it to what the
   bot serves. Any difference fails the build.

A dedicated adversarial pass tested typos, mixed-language input, prompt injection and
role-play, empty and huge inputs, emoji, and database-style input. Two real defects were
found and fixed during that pass. After the fixes, all 44 prices match their source PDFs
and no adversarial input could produce a wrong or invented price.
